    Dodavanje izmena u repozitorijum

    Da bismo rekli git-u da nam napravi snapshot trenutnih izmena, moramo komitovati te izmene. Arhitektura git procesa je takva, da umesto da git-u kažemo direktno da napravi snapshot, mi prvo dodamo fajlove u njegov indeks, pa ih onda komitujemo iz indeksa. Ovo može biti korisno ako smo recimo izmenili više fajlova, ali želimo snapshot samo nekih od njih. Kako se snapšot pravi od svih fajlova u indexu, to onda možemo postići deimičan snapshot ako samo željene fajlove dodamo u git index.

    Ove korake ponavljamo svaki put kad nešto konkretno izmenimo. Git prati naš folder, i u svakom trenutku zna koji fajlovi su izmenjeni u odnosu na njegov poslednji snapshot, ali neće praviti novi snapshot sam od sebe. Ovo se razlikuje od recimo MS Worda, ili vašeg tekstualnog editora, koji pravi snapshotove posle svakog ukucanog slova, tako da u njima možete da undo-ujete svako slovo koje ste ukucali. Kod git-a možete da undo-ujete samo od jednog snapshot-a do bilo kog prtehodnog.

    Dodavanje fajlova: $ git add *.*

    Da bismo dodali fajlove u indeks:

    MINGW32:~

    $ git add .

    $_

    Tačka iza reči add, znači da hoćemo da dodamo sve izmene. Ako bismo želeli da limitiramo git da doda samo jedan fajl, morali bismo precizirati naziv tog fajla umesto tačke. U većini slučajeva, $ git add . je sasvim zadovoljavajuće

    MINGW32:~

    $ git add "Moj fajl.txt"

    $ git add Moj_fajl.txt

    $_

    Napomena u vezi navodnika: Ne morate da ih koristite ako je ime vašeg fajla kao jedna reč.

    Kada uradimo git add na nekom fajlu, on je dodat u index, i od tog trenutka git počinje da prati sve izmene na njemu.

    Brisanje fajla već dodatog u index

    Ako ste greškom dodali fajl pomoću git add . komande, i ne želite da ga git prati, možete ga naknadno ukloniti, pomoću git rm --cached "pogrešan fajl.txt". Analogno, greškom dodat folder, npr u node.js-u ne želite da komitujete node_modules folder jer je ogroman, a lako ga je restorovati ako se obriše, pisali biste git rm --cached ime_foldera/ Da bi ovo upalilo, potrebno je takve fajlove i foldere dodati u .gitignore fajl, o kome će kasnije biti više reči.

    Komitovanje izmena: $ git commit

    Nakon što smo dodali fajl u indeks, možemo da ga komitujemo, i ovim će Git uslikati trenutnu situaciju u našem folderu. Ovo radimo na sledeći način:

    MINGW32:~

    $ git commit -m "moj prvi komit"
    [master (root-commit) 41f45e1] Moj prvi komit
    1 file changed, 1 insertion(+)
    create mode 100547 ime_fajla.txt

    $_

    Oznaka -m znači da komitujemo ove izmene, sa porukom (m od message), a poruka sledi u navodnicima, i u ovom slučaju glasi "moj prvi komit". Poenta poruke jeste da u par reči objasni šta je izmenjeno, tako da sutra ako nam zatreba stara verzija, znamo tačno gde da je nađemo. Dakle, poruka je kao kad smo svojevremeno čuvali verzije po dodatnim fajlovima, pa im davali nazive tako da znamo po čemu se razlikuju. Npr "dodato zaglavlje", "promenjena boja pozadine" i sl

    Ostatak ispisa koji nam Git izbacuje će se menjati od slučaja od slučaja. Ako komitujemo više fajlova, pisaće "x files changed", ako smo načinili veći broj izmena, pisaće "x insertions(+)" a ako smo neke redove obrisali, "x deletions(-)". Git će nam prijavljivati brisanje/dodavanje, čak iako nismo ništa obrisali samo smo nešto izmenili. Ako na primer zamenimo samo jedan zarez, git će smatrati da smo taj ceo red obrisali, i zamenili ga sa potpuno novim redom, koji se sasvim slučajno od prethodnog razlikuje samo po zarezu.

    Uobičajeno je da se komit poruke pišu u jednom redu i da imaju do 50 karaktera.
    • Možemo ako nam je to potrebno, napisati i dužu poruku, u više redova. Da bismo dodali poruku u više redova, morali bismo koristiti tekstualni editor. U tom slučaju, jedan red poruke ne treba da bude duži od 72 karaktera.
    • Praksa je da se poruke pišu u sadašnjem vremenu, kao šta ovaj komit radi, ne šta smo mi uradili s ovim komitom ili šta tek treba da uradimo. Komit poruke su o izmenama, ne o nama.
    • Poruka može da sadrži listu izmena, u kom slučaju bismo koristili * ili -.
    • Ako krenete da radite u nekoj firmi koja koristi git, moguće je da ćete dodavati i broj tiketa, u kome je opisan bag koji ste ovim komitom ispravili. Moguće su i druge oznake koje bi firma smatrala ilustrativnim za problem.

    Kada jednom komitujemo novu verziju fajla, vredi napomenuti da u našem folderu imamo samo poslednju verziju, u indeksu postoji samo poslednja verzija, ali u repozitorijumu imamo poslednju verziju, i sve izmene koje su do te poslednje verzije dovele, u vidu svih redova koje bi Git trebao da doda, ili oduzme, da bi od naše trenutne verzije, proizveo bilo koju prethodnu.

    Svaki put kada smo napravili izmene koje želimo da snapšotujemo, radimo iznova prvo git add . pa git commit -m "opis komita"

    Objedinjeni komit $ git commit -am

    Ukoliko želite, git add, i git commit se mogu objediniti u jednu komandu:

    MINGW32:~

    $ git commit -am "opis komita"
    [master (root-commit) 41f45e1] opis komita
    1 file changed, 1 insertion(+)
    create mode 100547 ime_fajla.txt

    $_

    Oznaka -a znači da ovom komandom dodajemo sve fajlove u git, a ostatak je istovetan prethodnoj commit komandi.

    Važno je napomenuti da će skraćena komanda git commit -am "poruka" raditi samo sa fajlovima koje smo nekada ranije dodali u index pomoću git add ime.fajla komand.e Ako smo u poslednjem koraku dodali novi fajl ili folder, tj, nikada ranije nismo uradili git add nad tim fajlom ili folderom, git commit -am "poruka" se neće primeniti na takve fajlove / foldere. Moramo ih prvo git addovati, pa onda nastaviti dalje

    Ova objedinjena komanda, updejtuje i indeks, i repozitorijum na poslednju verziju.

    Kod ovog pristupa treba samo imati na umu da neće obuhvatiti novo dodate fajlove, niti će ukloniti obrisane fajlove.
