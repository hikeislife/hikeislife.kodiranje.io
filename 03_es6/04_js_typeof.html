    typeof specijalnih tipova

    Pre nego što obradimo operator koji prima samo jedan operandunarni operator typeof, pokrićemo još dva typa podataka u js-u, koji se nazivaju zajedničkim imenom, "specijalni tipovi podataka", i to su:

    • Null
      •
    • Undefined
      •

    Kliknite ovde za typeof

    Null

    Ovaj specijalni tip podatka može imati samo jednu vrednost, i to null. Null znači da nema vrednosti. Ova vrednost nije analog 0 ili praznom stringu, jer su to dve konkretne vrednosti. Varijabli se može eksplicitno dodeliti null, odnosno vrednost joj se može obrisati.

    Undefined

    I ovaj specijalni tip podatka može imati samo jednu vrednost, i to undefined. Ako se neka promenjiva definiše, ali joj se ne dodeli vrednost, njena vrednost će biti undefined. Ovo će biti slučaj i ako se varijabla deklariše i dodeli joj se vrednost, ali se poziva pre nego što joj je vrednost dodeljena.

    let a;

    Ne treba brkati undefined sa "x is not defined". Prva znači da varijabla postoji, i vrednost joj je undefined, druga znači da varijabla ne postoji u posmatranom opsegu.

    U starijim brauzerima je bilo moguće dodeliti vrednost undefined-u, ali to sada više nije slučaj.

    U čemu je razlika između null i undefined zapravo?

    Jezici uglavnom nemaju po dva specijalna tipa podataka kao što su null i undefined. JS je po tome redak. Neki smatraju da bi jednu trebalo ukinuti, po mogućstvu undefined jer je null kraće za pisanje. Međutim undefined ima svoju svrhu.

    null i undefined zvuče zbunjujuće slično, zar ne? Ipak, nisu isti. Na prvom mestu, undefined znači da je promenjiva Dato joj je imedeklarisana, ali joj još uvek nije zadata vrednost, dakle nije definisana: undefined. S druge strane, null je dodeljena vrednost. Jedino što ta dodeljena vrednost znači; da nema vrednosti.

    U JS-u se znak jednakosti koristi da se promenjivoj dodeli vrednost:

    let mojaVarijabla = null;

    Dupli znak jednakosti == se koristi da se proveri da li su vrednosti dve promenjive jednake

    alert(null == undefined); // vraća true
    alert(2 == "2"); // vraća true

    Ove dve evaluacije vraćaju true.

    Trostruki znak jednakosti === se koristi za identičku evaluaciju, koja ne upoređuje samo vrednosti, nego i tipove:

    alert(null === undefined); // vraća false
    alert(2 === "2"); // vraća false

    Obe ove evaluacije vraćaju false, jer tipovi podataka nisu isti.

    Takođe, ako biste pokušali da zadate neku vrednost tipu null, dobili biste grešku reference:

    null = "neka moja vrednost"; // daje "ReferenceError"

    I prijavljuje nepravilnost u dodeli vrednosti s leve strane, to će reći da je null-u, nemoguće dodeliti vrednost. Međutim, isto ne važi i za undefined:

    udefined = "neka moja vrednost";

    Ne izbacuje nikakvu grešku ali i ne dodeljuje vrednost stringa promenjivoj undefined. Čak i nakon ovakve dodele, undefined ostaje undefined.

    Mada je moguće podesiti vrednost promenjive na undefined, to treba izbegavati. Ako vam je potrebna prazna promenjiva, podesite je na null, i pustite undefined da radi svoj posao neometano. Undefined treba da ostane slobodno da vam JS može skrenuti pažnju da ste neku varijablu definisali na pogrešnom mestu, ili je niste ni definisali.

    I konačno, razlika između null i undefined, nije broj:

    null - undefined
    > NaN

    Mala šala da razbije led (koji led? ima leda?)

    Znak - označava operaciju oduzimanja na kakvu smo navikli u stvarnom životu. Kako ni null ni undefined ne mogu da se evaluiraju kao brojevi, to ih je nemoguće oduzeti jednog od drugog. Otuda NaN, Not a Number. Ako pokušate od undefined da oduzmete neki konkretan broj, takođe ćete dobiti NaN, jer se undefined ne može konvertovati u broj. Ali ako od null pokušate oduzeti neki broj, dobićete taj broj, samo negativan. Zašto to? Null je odsustvo vrednsoti, tako da sve što ostaje, jeste znak minus i broj. Minus može biti unarni operator, odnosno služiti kao znak u negativnim brojevima, pa JS tako i evaluira ovu operaciju.

    operator typeof

    Operator typeof vraća tip promenjive. Pošto JS nije type safe jezik, moguće je da ćete nekad imati neku promenjivu koja će imati drugačiji tip nego što vi očekujete, na primer kada uzimate unos korisnika iz input polja, taj unos će biti string, ako vi očekujete broj, možete imati problema, pa u takvim situacijama možete utvrditi šta se dešava pomoću ovog operatora.

    Pošto typeof nije funkcija, ne piše se kao typeof(x);, već naprosto kao typeof x;.

    typeof "Pera Mitić"; // vraća string
    typeof "2"; // vraća string
    typeof 2; // vraća number
    typeof NaN; // vraća number, idi kići JS, pijan si!
    typeof true; // vraća boolean

    typeof "false"; // vraća string
    typeof undefined; // vraća undefined
    typeof null; // vraća object

    Aha?!? Šta je ovo, typeof null je objekat? Zašto ne null? Specifikacija JS-a kaže da je null tip, zašto onda typeof vraća objekat?

    Čestitam! Pronašli ste grešku u JavaScriptu. Tj, ovo je opšte prihvaćeno kao greška u JavaScriptu. Naravno, kao što to biva u svetu programiranja, postoje i oni koji misle da ovo nije greška i da ima savršenog smisla.

    U prvoj implementaciji JS-a, vrednosti su bile predstavljane kao 32-o bitne jedinice, koje su se sastojale od same vrednosti i na kraju kratkog "taga tipa", dužine 1-3 bita. Tagovi tipova su bili:

    • 000: Objekat
    • 1  : 31-o bitni intiger sa znakom (+ ili -)
    • 010: Double
    • 100: String
    • 110: Bulian

    Ako je poslednji bit bio 0, onda je tag bio 3 bita dug, a ako je poslednji bit bio 1, onda je tag imao 1 bit.

    Dve vrednosti su bile specijalne:

    • undefined(JSVAL_VOID), koji je predstavljao intidžer -230 što je broj koji pada izvan opsega intidžera, i
    • null(JSVAL_NULL) koji je imao vrednost takođe NULL Reference, ima vrednost koja govori da pointer ne ukazuje na validan objekat. Null Pointer se često koristi da ukaže recimo na kraj liste nepoznate dužine, ili neuspeh izvršavanja neke radnjeNULL pointer-a u mašinskom kôdu 0x00, dok mu je tag tipa, koji dolazi na kraj 32-o bitne jedinice, bio nula, dakle jedinstven tag tipa, za jedinstven tip, iliti sve zajedno: 0x000 što je takođe referenca da je vrednost 0 + tag objekta.

    U tim ranim danima, typeof je mislio da je null objekat, jer kad bi mu proverio tag tipa, video bi tag objekta.

    Pošto je JS nastao za samo 10 dana, ova greška nije bila otklonjena na vreme, a onda se pri standardizaciji JS-a, microsoft potrudio da spreči bilo kakve ispravke. Danas se ova greška održava, radi kompatibilnosti sa starim kodovima.

    E sad kad je sve to rečeno, vratimo se i na to zašto neki programeri misle da typeof null nije greška i da to tako treba da bude? Prilično je uobičajeno da null predstavlja prazne objekte. ECMAScript specifikacija definiše null kao primitivnu vrednost koja reprezentuje namerno odsustvo bilo kakve vrednosti objekta. Takođe, kako ECMAScript specifikacija kaže da typeof null treba da bude objekat, onda to deluje kao da je tako namerno. Samo što do sad svi znamo da je ECMAScript standard samo zakivanje grešaka.

    Sam Ajk, ne samo da ne poriče da je u pitanju greška, nego i rezignirano govori da je odustao od ideje da će se ista ikad ispraviti. Previše kôda se oslanja na taj bug kao na svojstvo jezika, da bi ispraviti je, značilo kreiranje mnogo, mnogo novih bagova.

    Pre implementacije inovacija ES6 (ES2015), typeof nikada nije davao greške. U najgorem slučaju bi davao da je neka varijabla, undefined:

    typeof a;
    var a = 1;

    Vraća undefined, zato što je varijabla var definisana nakon što je typeof pozvan. Ovo bi trebalo da izbaci grešku, ali zbog en: Hoisting - Dizanje, kao "dizanje zastave"hojstovanja o kome će kasnije biti više reči, JavaScript zna da varijabla postoji, ali ne zna koja joj je vrednost, pa joj daje vrednost undefined, kojoj je typeof, undefined.

    ES6 uvodi dve nove varijable, let i const, koje imaju životni vek vezan za blok kôda u kome se koriste, i tako se ne hojstuju, tako da ako sada napišemo:

    typeof a;
    let a = 1;

    Dobićemo ReferenceError, uz objašnjenje da se lexičkoj deklaraciji ne može pristupiti pre inicijalizacije (dodele vrednosti).

    A zašto je nije broj broj? typeof NaN === number

    Ovo zvuči kao greška zar ne? E pa nije. NaN je numerička vrednost, kao što su i brojevi, ili Infinity (rezultat deljenja nulom) i typeof oba, i Infinity i NaN, vraća da je u pitanju broj. Ali oba predstavljaju brojeve koje ne možete baš da kvantificirate. Kakav broj je ∞? Nije broj, ali jeste numerička vrednost. NaN je zapravo broj, koji se nalazi van opsega koje kompjuter prepoznaje, pošto se brojevi prezentuju kao niz bitova (nula i jedinica) zadate dužine, recimo 32 bita. Jedan bit odlazi na predznak, znači najveći broj koji se može prikazati i koga kompjuter može razumeti je 232 za intidžere bez predznaka. Svaki broj izvan tog opsega bi kompjuteru delovao kao besmislica koja baš liči na broj, dakle NaN.

    NaN je definisan kao broj koji nije jednak sam sebi:

    NaN == NaN // vraća false

    Ovo se navodi kao jedini pouzdan način da se utvrdi da li je nešto NaN. Ako nije jednako samom sebi, onda jeste NaN. Postoji i funkcija isNaN() koja vraća bulian, ali ova funkcija bukvalno proverava samo da li nešto jeste, ili nije broj, i vraća true i kada se evaluira string, ili objekat, jer oni zaista nisu brojevi, tako da nije vezana za NaN, jer typeof stringova i objekata nije number, što jeste slučaj sa NaN.

    Ironično, isto ne važi za Infinity:

    Infinity == Infinity // vraća true
    Infinity == 5 * Infinity // vraća true
    Infinity === Infinity // vraća true
    Infinity === 5 * Infinity // vraća true

    Ovo onda u celosti zanemaruje Kantorov dijagonalni dokaz po kom beskonačni niz realnih brojeva može samo da bude veći od beskonačnog niza celih brojeva.
